# qa-kit — Claude Code plugin for QC

Generates test cases from **spec / DD / DB definition** → runs them → triages → reports.
Follows **ISTQB/JSTQB** (vocabulary, phase framework) + **VSTeP** (テスト観点, test viewpoints).

Does not read source code. Not unit testing. Not a developer's view.

## Install

```bash
/plugin marketplace add Sotatek-TuanNguyen20/qa-kit-plugin
/plugin install qa-kit@qa-internal
```

## Usage

```bash
mkdir qa-project-a && cd qa-project-a
claude
> /qa-kit:init project-a          # scaffold once
# comtor drops docs into docs/, get DDL from dev into db/
> /qa-kit:design login            # → testcases/login.yaml + gap + coverage
# HUMAN REVIEW → merge
> /qa-kit:run login --build=v1.2.3
> /qa-kit:eval login              # triage → routing
> /qa-kit:report login
# dev fixes
> /qa-kit:retest login v1.2.4     # scope = retest_failed + REGRESSION
```

## Plugin ↔ project boundary

|                                   | Plugin (read-only) | Project repo               |
| --------------------------------- | ------------------- | --------------------------- |
| commands, skills, tools, schemas  | ✅                   |                              |
| `context/viewpoints.md` (shared)  | ✅                   |                              |
| `context/conventions.md`          | ✅                   |                              |
| CLAUDE.md, .claude/settings.json  | template             | ✅ generated by `/qa-kit:init`   |
| `context/viewpoints-local.md`     |                      | ✅ project-specific 観点     |
| docs/ db/ testcases/ results/     |                      | ✅                           |

**The plugin CANNOT ship CLAUDE.md and settings.json** — that's why `/qa-kit:init` exists.
Skipping `/qa-kit:init` means you get the commands but none of the guardrails.

## Two layers of viewpoints

```
${CLAUDE_PLUGIN_ROOT}/context/viewpoints.md   ← shared, every project, edited via PR
./context/viewpoints-local.md                 ← project-specific
```

A local 観点 that catches a real bug ≥2 times → **PR-promote it into the plugin**. Every project benefits.
Not promoting it means the experience dies with that one project.

## 5 non-negotiable rules

1. **Conditions may go beyond the doc. Expected results may not.**
2. **`evidence` is immutable** — blocks editing expected to match actual.
3. **Unsure why a test failed → default to BUG**, not "bad test case."
4. **Retest ≠ failed cases.** Must include a regression group.
5. **blocked ≠ fail.** High pass rate with many blocked cases = false optimism.

## Metric

Merge-without-edit rate of test cases, per module. <60% → feed more into `viewpoints.md`,
**don't** edit the prompt or add a skill.
